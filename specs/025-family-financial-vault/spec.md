# Feature Specification: Family Financial Vault (RR-only)

**Feature Branch**: `025-family-financial-vault`
**Created**: 2026-05-04
**Status**: Decisions Locked 2026-05-04 — ready for `/speckit-plan` once feature 024 merges to `main`
**Input**: User description: "Another HTML file linked to the RR dashboard. Local-only. Two purposes: (a) ongoing monthly financial check-in for both spouses; (b) if Roger passes away, Rebecca opens it and learns what accounts exist and the recommended strategy to receive each one (e.g., 401(k) at UPPAbaby → contact HR → spread withdrawals to avoid top tax bracket). Includes a Claude-API chatbox locked to the embedded financial info."

> **Scope note:** This feature is RR-only. Generic dashboard is **NOT** in scope. The new file (`FIRE-Family-Vault-RR.html`) sits beside `FIRE-Dashboard.html` and shares its dark-theme CSS variables but is otherwise an independent single-file HTML app. The existing two-files-in-lockstep rule (CLAUDE.md "Default rule") does **NOT** apply.
>
> **Privacy posture:** Every dollar value, account number, SSN, beneficiary name, and policy reference exists only in the user's browser localStorage on a single laptop. Nothing in this spec, nor in any committed file, contains real PII. The user populates the data the first time they open the page; updates happen monthly. The only outbound network call is to `api.anthropic.com` when the user types a question into the chatbox AND has pasted a Claude API key.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Account inventory CRUD (P1)

Roger or Rebecca opens `FIRE-Family-Vault-RR.html` for the first time and walks a setup wizard that captures every account the family owns. The schema covers: bank, brokerage, retirement (401(k), IRA, Roth), insurance with cash value (Nan Shan), REIT shares (Ark7), foreign bank accounts (FBAR-reportable), joint-tenancy accounts, and one-off "miscellaneous" assets (e.g., HSA, crypto). Each account captures owner, institution, type, country, last-4 of account number (full number optional and stored locally only), institution contact info, login portal URL, and current balance with timestamp. Edit and delete are inline. The data persists in `localStorage` under the namespace `vault.rr.v1`.

**Why this priority**: Nothing else in the feature works without an inventory. This is the foundation.

**Independent Test**: User opens the file with empty localStorage. Adds 3 accounts (one bank, one brokerage, one foreign bank). Refreshes. All 3 accounts appear with the same data. Edits one. Refreshes. Edit persists. Deletes one. Refreshes. Only 2 remain.

**Acceptance Scenarios**:

1. **Given** a fresh `localStorage`, **When** Rebecca opens the file, **Then** the page shows a "Set up your vault" empty state with a single CTA to add the first account.
2. **Given** at least 1 account exists, **When** the user clicks "Add Account", **Then** a modal captures all schema fields and validates required ones (owner, institution, type, country) before save.
3. **Given** the Roger Hsu / Rebecca Chu / Joint owner toggle, **When** the user filters by owner, **Then** only matching accounts are visible and the count badge updates.
4. **Given** the user filters by country (US / Taiwan / China / Other), **When** the filter is applied, **Then** account counts and grand-total USD value reflect the filter.

### User Story 2 — Estate procedure playbook (P1)

For each account in the inventory, a "What happens if [Roger / Rebecca] passes away" panel renders rule-based procedural text from a pre-authored knowledge base (NOT generated by an LLM). The knowledge base is a static JSON file `vault-procedures.json` keyed by account category. Categories include:

- **`employer-401k-roth`** → "Surviving spouse contacts UPPAbaby HR (Monahan Products LLC, [phone]). Roger's beneficiary on file should be Rebecca. Custodian will guide a spousal rollover to either a (i) Spousal Inherited Roth IRA or (ii) Rebecca's own Roth IRA. **Strategy guidance:** Avoid lump-sum withdrawal — even a Roth-401(k) lump sum can push Rebecca into AMT or affect ACA / IRMAA cliffs. Spread withdrawals over 10 years (post-SECURE-Act inherited-account rules)."
- **`employer-401k-trad`** → ordinary-income tax warning, 10-year stretch math, RMD handling.
- **`us-brokerage-tod`** → Webull / Schwab transfer-on-death paperwork, basis step-up at date of death.
- **`us-brokerage-jtwros`** → joint tenancy with rights of survivorship — automatic.
- **`us-bank-joint`** → East West Bank — automatic.
- **`foreign-bank-taiwan`** → Apostille requirement for U.S. death certificate, translation, Taiwan inheritance tax exemption thresholds, repatriation guidance.
- **`foreign-bank-china`** → tighter capital controls; specific Bank of China + China Construction Bank inheritance procedures; recommend retaining a Taiwan-licensed CPA for cross-border estate.
- **`taiwan-life-insurance`** → Nan Shan Life Insurance policy claim procedure, beneficiary review.
- **`reit-ark7`** → Ark7 share inheritance; whether SDIRA or taxable; transfer process.

Each procedure entry includes: who to contact, expected timeline, paperwork list, tax-strategy note, and a "Common mistakes" subsection.

**Why this priority**: This is the load-bearing user-facing value. Without this, Rebecca opens the file at the worst possible moment and gets nothing useful.

**Independent Test**: User clicks an account of category `employer-401k-roth`. The procedure panel renders within 200ms with the full procedure for that category. Switching account categories shows the correct procedure for each. The procedure text is static — opening the file with the network disabled still shows full procedures.

**Acceptance Scenarios**:

1. **Given** Roger has an account categorized `employer-401k-roth`, **When** Rebecca clicks "Death procedure" on that account, **Then** the rendered panel shows the UPPAbaby HR contact path AND the spread-over-10-years tax-strategy paragraph.
2. **Given** the network is offline, **When** the user views any procedure, **Then** the procedure renders normally (knowledge base is bundled, not fetched).
3. **Given** an account has no matching procedure category (user typed a freeform "Other" category), **When** the user requests the procedure, **Then** the page renders a "Generic procedure: contact a probate attorney" fallback with no broken UI.

### User Story 3 — Tax-optimized inherited-401(k) withdrawal calculator (P1)

For each retirement account (`employer-401k-trad`, `traditional-ira`), the procedure panel includes an interactive **"What if I take it out" calculator**. Inputs: starting balance (auto-filled from inventory), Rebecca's current age, Rebecca's expected non-retirement annual income post-Roger (W-2, SS, etc.), filing status (likely single from year of death + 2 years), state of residence (default MA). Output: a 10-year withdrawal schedule that minimizes total federal + MA tax over the stretch period, computed by reusing the existing **bracket-fill-smoothed** strategy from the FIRE calculator's [`calc/withdrawal-strategies.js`](../../calc/withdrawal-strategies.js).

Three columns are always shown side-by-side for comparison:

| Strategy | Year 1 withdrawal | Year 1 federal tax | Year 1 MA tax | 10-yr total tax | 10-yr ending balance |
|---|---|---|---|---|---|
| Lump sum (year 1) | full balance | top bracket → IRMAA → AMT | 5% MA flat | (highest) | $0 |
| Even tenths | balance / 10 | computed | computed | (medium) | $0 |
| Bracket-fill smoothed | computed | bracket-cap | computed | (lowest) | $0 |

This is the concrete advisory feature Roger asked for. Re-uses calc engine code from [`calc/withdrawal-strategies.js`](../../calc/withdrawal-strategies.js) (or a thin import — see Plan).

**Why this priority**: This is the tangible reason Rebecca wouldn't just call a random CPA. Numbers make the strategy concrete.

**Independent Test**: Given a $500K traditional 401(k) and Rebecca's expected income of $60K, the calculator should produce a 10-year schedule where each year's combined income (Rebecca's $60K + that year's withdrawal) caps at the 22% federal bracket boundary, with any residual deferred. Total 10-year tax should be at least 30% lower than the lump-sum scenario.

**Acceptance Scenarios**:

1. **Given** a traditional 401(k) account, **When** the user opens the calculator and enters Rebecca's expected income, **Then** all three strategies render with year-by-year breakdowns and a "Recommended: Bracket-fill" callout on the lowest-tax row.
2. **Given** a Roth 401(k) account, **When** the user opens the same calculator, **Then** the federal tax columns show $0 (qualified distributions tax-free) AND the calculator surfaces all three cliff warnings — IRMAA Medicare premium surcharge, ACA premium tax credit cliff, AMT — if Rebecca takes a large slice in any one year (per user decision 2026-05-04: surface all cliffs even on Roth, since they all key off MAGI which Roth distributions affect).
3. **Given** the user changes Rebecca's expected income, **When** they hit Recompute, **Then** the schedule updates and the "Recommended" callout may move to a different row.
4. **Given** a printable-summary button, **When** clicked, **Then** the strategy table renders as a single page that prints cleanly (margins, no chart cropping).

### User Story 4 — Recurring obligations calendar (P2)

The vault tracks the family's annual recurring tax / regulatory obligations and shows a 12-month forward-looking calendar widget. Pre-populated obligations:

- **April 15** — Federal 1040 + Form 8938 FATCA + MA Form 1 + Q1 1040-ES + FBAR (BOTH spouses must file separately)
- **June 15** — Q2 1040-ES
- **September 15** — Q3 1040-ES
- **October 15** — FBAR auto-extension deadline (if missed April)
- **January 15** — Q4 1040-ES

Each obligation has: title, due date, who must act (Roger / Rebecca / Both), pre-authored "How to do it" link to the appropriate procedure entry (FBAR procedure → BSA E-Filing walkthrough; 1040-ES → IRS Direct Pay walkthrough), and a "Mark done for [year]" checkbox stored in `vault.rr.v1.obligations[year]`. Past-due unchecked obligations show in red.

This list is critical because Rebecca alone (post-Roger) still has to file her own FBAR if her foreign bank balances exceed $10,000 aggregate — which they currently do.

**Why this priority**: Calendar discipline survives Roger's absence. Critical for keeping FBAR / FATCA compliant during a year Rebecca is grieving.

**Independent Test**: Open the vault on April 1. The calendar shows "April 15: 1040, 8938, FBAR Roger, FBAR Rebecca, MA Form 1, Q1 1040-ES" with an "act in 14 days" badge. Mark "FBAR Roger" done. The badge updates and the obligation moves to the "Done this year" pile.

**Acceptance Scenarios**:

1. **Given** the current date is within 30 days of any obligation, **When** the calendar renders, **Then** that obligation shows a yellow "approaching" badge.
2. **Given** the date is past an unchecked obligation, **When** the calendar renders, **Then** the obligation shows a red "overdue" badge AND a separate banner at the top of the page surfaces it.
3. **Given** Rebecca is the sole-living user and Roger's accounts are tagged `roger-deceased=true`, **When** the calendar renders, **Then** Roger's FBAR obligation is marked "Final FBAR for Roger (year of death)" with a procedure link to "Filing FBAR for a deceased spouse".

### User Story 5 — Claude API chatbox with locked context (P2)

A right-side panel hosts a chatbox. The first time the user opens it, a setup form asks for a Claude API key (stored in `localStorage.vault.rr.v1.apiKey` — never committed, never sent anywhere except `api.anthropic.com`). The chatbox uses the **Messages API** with a system prompt that:

1. **Embeds the full vault JSON** (accounts, procedures, obligations, but NOT the API key itself) as the model's only source of factual context.
2. **Restricts scope**: "You may answer ONLY questions whose answer is grounded in the JSON above. If the user asks anything outside this scope — market predictions, specific fund recommendations, legal advice, tax advice not pre-authored in the procedures, gossip, general knowledge — politely refuse and recommend they consult a licensed professional."
3. **Forbids hallucination**: "Do not invent account numbers, balances, beneficiaries, or institutions not present in the JSON."
4. **Requires citation**: "Every factual claim must reference the account ID or procedure category from the JSON."
5. **Defaults to caution**: "When unsure, say 'I don't have that information in your vault. Please add it via the inventory page or consult [appropriate professional].'"

The user can click "Show system prompt" to inspect what's being sent. Default model: `claude-haiku-4-5-20251001` (cheap, fast, sufficient for retrieval-style Q&A); user can switch to `claude-sonnet-4-6` or `claude-opus-4-7` from a dropdown. Prompt caching is enabled on the system block to amortize the embedded JSON across the conversation.

A persistent disclaimer above the chatbox: **"This chatbox echoes information from your local vault. It is not a substitute for a CPA, estate attorney, or financial advisor. For any consequential decision (taxes, withdrawals, beneficiary changes), call a licensed professional."**

**Why this priority**: This is the part Roger explicitly asked for, but the playbook + calculator carry most of the actual value, so this is P2 not P1.

**Independent Test**:
- Question 1 ("What accounts do I have at E. Sun Bank?") → returns a structured list grounded in the JSON.
- Question 2 ("Should I sell my TSLA?") → refuses, points to a CPA / fiduciary advisor.
- Question 3 ("What is the capital of France?") → refuses, says it's out of scope.
- Question 4 (asks for an account that doesn't exist) → says "I don't see that account in your vault."

**Acceptance Scenarios**:

1. **Given** no API key is set, **When** the user opens the chatbox, **Then** they see a setup form that explains where to get a key (https://console.anthropic.com/settings/keys) and the privacy posture.
2. **Given** the API key is set, **When** the user asks a question grounded in the vault, **Then** the answer cites at least one account ID or procedure category from the JSON.
3. **Given** the user asks an out-of-scope question, **When** the model responds, **Then** the response is a short, polite refusal that names the right kind of professional to consult.
4. **Given** the user clicks "Reset key", **When** confirmed, **Then** `localStorage.vault.rr.v1.apiKey` is deleted and the chatbox returns to the setup form.
5. **Given** the user clicks "Disable chatbox", **When** confirmed, **Then** the chatbox UI is hidden and the rest of the vault works fully without any network calls.

### User Story 6 — Monthly snapshot history (P2)

Once a month, the user clicks "Update balances" and walks through every account, refreshing the current balance. The previous balance is automatically appended to that account's `history[]` array as `{ date, balanceUSD, source: 'manual' }`. A summary screen shows: total net worth this month vs. last month, biggest movers, accounts not updated in > 60 days (flagged for refresh), and a one-click "export this month's snapshot to CSV" matching the format of the existing [`FIRE-snapshots.csv`](../../FIRE-snapshots.csv) so Roger can append the row to that file if he wants.

**Why this priority**: Keeps the inventory current. Without it the death-procedure data ages out and becomes wrong.

**Independent Test**: User updates 3 of 5 accounts. The summary screen flags the 2 unupdated accounts. The history array on each updated account grows by one entry. CSV export downloads a single-row CSV with all 5 accounts' current balances.

**Acceptance Scenarios**:

1. **Given** the user opens the vault, **When** the inventory list renders, **Then** each account shows a passive "Last updated: N days ago" caption — informational only, NO banner, NO modal, NO nag (per user decision 2026-05-04: "I will update every month, don't bother to remind").
2. **Given** the user finishes a monthly update, **When** they view an account's detail, **Then** a "Last 12 months" mini-chart shows the balance history.
3. **Given** the CSV export, **When** the file downloads, **Then** the column ordering matches the existing FIRE-snapshots.csv schema (DB Engineer-owned).

### User Story 7 — Family / legal pointers (P2)

A dedicated "Critical Family Info" section captures non-account information that's load-bearing in an estate event:

- **Will**: location of the executed copy (e.g., "fireproof safe at home, code [in password vault]"), executor's name and contact, attorney's contact.
- **Power of Attorney** (financial + healthcare): who holds it, where the documents are stored.
- **Healthcare Directive / Living Will**: where stored, key wishes.
- **Guardianship plan for minor children**: nominated guardians (primary + alternate), why, contact info, financial provision plan.
- **Life insurance policies**: UPPAbaby group-term life (1× or 2× salary — verify with HR), any private term-life policies, beneficiary on each.
- **Password vault location**: which manager (1Password / Bitwarden / etc.), where the master password recovery instructions live.
- **Funeral / burial preferences**: short freeform note.
- **Important contacts**: CPA, lawyer, financial advisor, primary care physician, employer HR, life insurance agent.

Each entry is freeform text; the section is collapsible so it doesn't clutter the day-to-day account view.

**Why this priority**: Non-financial but critical. A complete vault without this is half-useful.

**Independent Test**: Open the vault. Click "Critical Family Info". Add an executor entry. Refresh. The entry persists.

**Acceptance Scenarios**:

1. **Given** the section is collapsed by default, **When** Rebecca opens the page in an estate event, **Then** an "If something has happened" CTA at the top pre-expands the Critical Family Info section.
2. **Given** the user adds a guardianship plan, **When** the chatbox is asked "Who are the kids' guardians?", **Then** it answers from the JSON, NOT from the LLM's general knowledge.

### User Story 8 — Import retirement totals from RR FIRE Dashboard (P3)

A one-click "Import from FIRE-Dashboard.html" button reads the RR dashboard's `localStorage` keys (under whatever namespace the RR dashboard uses — DB Engineer to confirm) and offers to map the aggregated retirement / brokerage / cash totals into the vault. Because the RR dashboard stores aggregates (e.g., "401(k) bucket = $X") and the vault stores per-account detail, the import is a one-way "here's the bucket total, please allocate it across the accounts you've listed". The vault never overwrites the FIRE dashboard's data.

**Why this priority**: Convenience. Reduces double-entry. Lower priority because the schemas don't align cleanly enough for an automatic mapping.

**Independent Test**: With the FIRE dashboard previously populated, the vault's import button offers a 4-bucket reconciliation form. The user assigns the $260K cash bucket across their actual cash accounts. Save persists. Re-import doesn't double-count.

**Acceptance Scenarios**:

1. **Given** the FIRE dashboard's localStorage is unset, **When** the user clicks Import, **Then** a friendly error explains they need to open the FIRE dashboard first.
2. **Given** the buckets are imported, **When** the user assigns 100% of each bucket to their accounts, **Then** vault account balances update and history rows are appended with `source: 'fire-dashboard-import'`.

### User Story 9 — Privacy architecture (P1)

The vault MUST satisfy the following privacy invariants, verified by a one-time audit:

1. **No outbound network calls** at page load, idle, or interaction time — except when the user submits a chatbox question (single POST to `api.anthropic.com`).
2. **No third-party scripts** of any kind (no Google Fonts, no analytics, no Chart.js CDN — vault is single-file pure CSS, optional inline SVG charts only).
3. **No telemetry**, no error reporting, no console output of PII.
4. **localStorage encryption (optional)**: a setup-time toggle prompts for a passphrase; if set, the vault payload is AES-GCM encrypted using a key derived via PBKDF2-SHA-256 (300k iterations) from the passphrase. The passphrase is held in memory only for the session.
5. **Print-to-PDF** mode generates a complete offline backup that the family can store on paper.
6. **Self-hosted file** — no `<script src="https://...">` tags. Everything inline.

**Why this priority**: This file contains the family's complete financial fingerprint. A privacy regression is a P1 incident.

**Independent Test**: Open the vault in DevTools → Network tab. Reload, click around, add accounts, view procedures. **Zero requests** in the Network tab. Type a chatbox question — exactly one POST to `api.anthropic.com`. Open the file in Notepad — every dependency is inline.

**Acceptance Scenarios**:

1. **Given** the user has not enabled the chatbox, **When** they use the vault for any task, **Then** the Network tab shows zero requests.
2. **Given** the user enables passphrase encryption, **When** they reload the page, **Then** the vault prompts for the passphrase and refuses to render the inventory until it's entered correctly.
3. **Given** an incorrect passphrase, **When** entered three times, **Then** the vault rate-limits subsequent attempts (1s → 10s → 60s) without permanently locking (so a forgotten passphrase isn't catastrophic — the user can wait and retry).

## Edge Cases

- **What if the family computer is shared with kids?** → Encryption recommended in Critical Family Info; the post-encryption page state shows nothing useful pre-passphrase.
- **What if Rebecca opens the file and Roger is incapacitated, not deceased?** → The "If something has happened" expand button has two sub-paths: "Roger is incapacitated" (POA path) and "Roger has passed away" (estate path). Different procedure links.
- **What if an account is held by Roger but beneficiary is the trust, not Rebecca?** → The Account schema has a `beneficiary` freeform field; procedures key off the *category*, not the beneficiary, so the procedure still renders. A "Beneficiary on file" callout flags Rebecca to coordinate with the trustee.
- **What if the Claude API key is rate-limited or invalid?** → Chatbox shows the API error verbatim, suggests checking key validity, never fails silently.
- **What if the user switches laptops?** → The vault has an "Export full backup" button (encrypted JSON). Import on the new laptop restores everything. The user should keep one printed PDF copy in the same fireproof safe as the will.
- **What if Rebecca dies first?** → Symmetric. All procedures are written from "the surviving spouse" perspective; categories cover Rebecca's accounts (Schwab, Taiwan banks, Nan Shan Life Insurance) just as thoroughly as Roger's.
- **What if the kids inherit (both parents die)?** → Out of scope for v1. v2 could add a "Both parents deceased" path that surfaces guardianship + UTMA/UGMA / 529 plans.

## Functional Requirements

### Core schema (FR-001 — FR-010)

- **FR-001**: System MUST store all data in `localStorage.vault.rr.v1` as a single JSON blob. Schema is versioned via the `v1` suffix.
- **FR-002**: System MUST support 8 account categories: `us-bank`, `us-brokerage-tod`, `us-brokerage-jtwros`, `employer-401k-roth`, `employer-401k-trad`, `traditional-ira` / `roth-ira`, `foreign-bank-taiwan`, `foreign-bank-china`, `taiwan-life-insurance`, `reit`, `other`.
- **FR-003**: System MUST allow per-account fields: `id` (UUID), `owner` (`roger` / `rebecca` / `joint`), `category`, `institution`, `accountNumberLast4`, `accountNumberFull` (optional), `currencyNative`, `currentBalanceUSD`, `lastUpdated` (ISO-8601), `loginUrl`, `contactPhone`, `contactEmail`, `beneficiary` (freeform), `notes` (freeform), `history[]` (array of `{date, balanceUSD, source}`).
- **FR-004**: System MUST persist a `vault.rr.v1.criticalFamilyInfo` section with subsections: will, POA, healthcareDirective, guardianship, lifeInsurance, passwordVault, funeral, contacts.
- **FR-005**: System MUST persist a `vault.rr.v1.obligations[year]` section keyed by 4-digit year, holding boolean done flags for each pre-defined recurring obligation.
- **FR-006**: System MUST persist a `vault.rr.v1.apiKey` (optional, only if the user enables the chatbox).
- **FR-007**: System MUST persist a `vault.rr.v1.encryption` block holding `{ enabled: bool, salt, iv }` if encryption is enabled. The actual passphrase is NEVER persisted.

### Procedure knowledge base (FR-011 — FR-015)

- **FR-011**: System MUST bundle a static `vault-procedures.json` inline (not fetched from network) keyed by account category.
- **FR-012**: Each procedure entry MUST contain: `whoToContact`, `expectedTimeline`, `paperworkChecklist[]`, `taxStrategyNotes`, `commonMistakes[]`.
- **FR-013**: System MUST render a "Generic procedure: contact a probate attorney" fallback for accounts in the `other` category.
- **FR-014**: System MUST display, on every procedure panel, a persistent disclaimer: "This procedure is general information, not legal or tax advice. Consult a licensed estate attorney and CPA before acting."
- **FR-015**: System MUST provide a "Print procedure" button on every panel that produces a clean single-page printout suitable for handing to a CPA or attorney.

### Withdrawal calculator (FR-021 — FR-025)

- **FR-021**: System MUST compute a 10-year inherited-account withdrawal schedule for each retirement account category (`employer-401k-trad`, `traditional-ira`, `employer-401k-roth`, `roth-ira`).
- **FR-022**: System MUST compare three strategies side-by-side: Lump-sum, Even-tenths, Bracket-fill-smoothed.
- **FR-023**: System MUST use 2026 federal MFJ → Single tax brackets (Rebecca's filing status post-year-of-death) and 5% MA flat rate.
- **FR-024**: System MUST flag IRMAA (>$103,000 single MAGI in 2026), ACA cliff (400% FPL), and AMT thresholds when any year's withdrawal pushes past them.
- **FR-025**: System SHOULD reuse `calc/withdrawal-strategies.js` from the FIRE dashboard if cleanly extractable; otherwise embed a simplified version.

### Chatbox (FR-031 — FR-038)

- **FR-031**: Chatbox MUST use Anthropic Messages API exclusively (no other LLM provider).
- **FR-032**: Chatbox system prompt MUST embed the full vault JSON (excluding `apiKey` and any `accountNumberFull` fields if the user toggles "redact account numbers in AI context").
- **FR-033**: Chatbox MUST use prompt caching on the system block (5-min TTL) to amortize the embedded JSON cost across multi-turn conversations.
- **FR-034**: Chatbox MUST refuse, with a polite static response, any of: market predictions, fund recommendations, legal opinions, tax opinions not in the bundled procedures, general-knowledge questions.
- **FR-035**: Chatbox MUST cite a vault account ID or procedure category for every factual claim.
- **FR-036**: Chatbox MUST default to `claude-opus-4-7` (deepest reasoning, per user decision 2026-05-04) and allow user-side switching to `claude-sonnet-4-6` and `claude-haiku-4-5-20251001` via a dropdown for cost-sensitive sessions. The current per-query cost (input tokens × Opus rate, with prompt caching applied to the embedded JSON system block) is shown next to the model dropdown so the user can see what each conversation costs.
- **FR-037**: Chatbox MUST support a "Show system prompt" debug toggle so the user can audit what's being sent.
- **FR-038**: Chatbox MUST be entirely disable-able. With the chatbox disabled, the vault MUST make zero network calls.

### Privacy (FR-041 — FR-046)

- **FR-041**: Vault HTML file MUST be self-contained (no external `<script src>` or `<link href>` tags pointing outside the file). Inline-only.
- **FR-042**: Vault MUST make zero outbound network calls except when the user submits a chatbox question.
- **FR-043**: Vault MUST offer optional AES-GCM-256 + PBKDF2-SHA-256 (300k iterations) localStorage encryption. Default is **OFF** (per user decision 2026-05-04) to keep day-to-day usability simple. The user can enable it from Settings at any time and is then prompted for a passphrase on every page load.
- **FR-044**: Vault MUST refuse to load console output containing any PII (account numbers, SSNs, balances). All console.* calls MUST log opaque IDs only.
- **FR-045**: Vault MUST provide an "Export encrypted backup" button producing a passphrase-protected file the family can store offsite.
- **FR-046**: Vault MUST provide a "Print full vault to PDF" button generating a paper backup formatted for a 3-ring binder.

### Recurring obligations (FR-051 — FR-053)

- **FR-051**: Vault MUST pre-populate the recurring obligations list with the following entries (verifiable against `2025_Tax_Filing_Guide_FATCA_FBAR.md`): Federal 1040, MA Form 1, FBAR (per spouse), Form 8938 FATCA, four 1040-ES quarterly estimates, charitable receipt aggregation reminder.
- **FR-052**: Vault MUST mark obligations done per year, never globally.
- **FR-053**: Vault MUST surface overdue and upcoming-30-day obligations in a top-of-page banner.

### Bilingual UI (FR-061 — FR-065)

- **FR-061**: Vault MUST support EN and zh-TW from v1, matching the existing FIRE-Dashboard.html locale toggle pattern (per user decision 2026-05-04).
- **FR-062**: Every UI string (labels, buttons, errors, banners, modal copy, settings) MUST live in a translation catalog with both EN and zh-TW values; no hard-coded strings outside the catalog.
- **FR-063**: The bundled `vault-procedures.json` MUST include both `en` and `zh-TW` versions of every procedure entry's `whoToContact`, `expectedTimeline`, `paperworkChecklist`, `taxStrategyNotes`, and `commonMistakes` fields.
- **FR-064**: The locale toggle MUST persist in `localStorage.vault.rr.v1.locale` and default to EN on first open. Toggling the locale MUST NOT mutate any user-entered data — only the rendered chrome.
- **FR-065**: The chatbox system prompt MUST detect the user's current locale and instruct the model to answer in that language. If the user types in the other language, the chatbox replies in whichever language the user typed in (override).

### Procedure knowledge base — UPPAbaby HR contact (FR-016)

- **FR-016**: The `employer-401k-roth` procedure entry MUST include explicit fields for `hrContactName`, `hrContactPhone`, and `hrContactEmail`. These render as TBD placeholders in the v1 deliverable; the user will provide values after spec sign-off and they get baked into the procedure JSON before merge (per user decision 2026-05-04).

```
Vault (root)
```

## Assumptions

- Both spouses are comfortable using a single-file HTML app; no installer / packaging needed.
- The Claude API key is the user's responsibility; we provide instructions but never auto-provision.
- Tax brackets, IRMAA thresholds, and ACA cliff numbers are baked in for tax year 2026 only; an annual update is in scope for v1.1.
- The user accepts that browser localStorage is not a forever-storage solution and will export an encrypted backup at least quarterly.
- The vault does not replace a CPA / estate attorney; it organizes information so those professionals can be hired effectively.

- Multi-device sync (no cloud — by design).
- Live account aggregation via Plaid or similar — would violate the privacy posture.
- Automated balance updates from bank screen-scraping — same reason.
- "Both parents deceased" guardianship walkthrough for the kids' inheritance (deferred to v2).
- Generic dashboard parity (RR-only feature).
- Languages other than EN and zh-TW (Spanish, Cantonese, etc. — deferred to v2 if ever needed). Bilingual EN + zh-TW IS in scope for v1 per user decision 2026-05-04.
- Crypto-asset inheritance (Roger doesn't currently hold crypto per the 2025 tax docs — if that changes, add a `crypto` category in a follow-up).
- Print/export backup reminders (per user decision 2026-05-04: user maintains the vault monthly themselves, no nag UI).
